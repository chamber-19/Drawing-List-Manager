Why this is the better pattern

Navigation matches the spec
  • Type-band tree on the left. The R3P numbering convention IS a hierarchy (discipline → type → band → seq). The nav reflects that 1:1 instead of flattening it into a search box. You click "Cable Tray Routing" the way you'd think it.
  • Status filters compose with the type tree. Click "Cable Tray Routing" and "In design" together — the workspace shows just the cable tray drawings still being designed. No filter modal, no faceted search UI. The whole left rail IS the filter.
  • Counts everywhere. Every node in the tree shows its count. Bands with zero drawings dim out. You see project shape at a glance.
Band cards keep standards alive
  • Each band is a card with band metadata in its header. Sequence range, label, fill progress bar, "n/m active" count. The band IS the visual unit, not just metadata on a row.
  • Next-free seqs are always visible. The band footer shows ↳ next free 0302-0304, 0306, 0309-0400. When you go to add a drawing you don't have to think — the answer is on screen. This single feature replaces the entire mental load of "what's the next free number" that drives most of the existing manual workflow.
  • "Add in this band" is contextual. The button knows the band already, so add-drawing flow becomes: pick a seq from the suggestions, type the description, done. No band selector, no type dropdown, no validation that you didn't pick a number outside the band.
Selection model is bulk-first
  • Inspector adapts to selection size. One drawing → details + history + files. Multiple drawings → summary of common/different fields + bulk actions. No separate "bulk edit mode" — it's just what the inspector does when more than one row is selected.
  • Bulk actions match real workflow. "Advance to next rev" with today's date, "Promote to IFC" (validates and starts numeric series at 0), "Mark superseded" (preserves audit trail per rule #3 in copilot-instructions: drawing numbers are immutable identifiers). These are operations you do on sets of drawings, not on individual ones.
  • Status glyphs visualize progression. ○ → ◐ → ◑ → ● is a quality-completion gradient that reads at a glance across 187 drawings. Color reinforces it: grey → amber → blue → green.
Reconcile is a first-class view
  • Three columns of asymmetric problems. Register-without-DWG (severe — drawings you owe), DWG-without-register (warn — orphans, often during refactors), PDF-stale (warn — the most common drift, since PDFs are exported less often than DWGs change). Different severities get different border colors so triage is visual.
  • Click any row to jump back to the workspace filtered to that drawing. Reconcile is a triage view that hands off to the workspace for fixing.
What this slice ships
  • Project marker → load → display the type-band tree (read-only) → render band cards with drawings (read-only) → single + bulk selection inspector (read-only) → reconcile tab. Project scan endpoint is required to make reconcile and file-presence work — that's the new backend work this slice introduces beyond what already exists.
  • Save / Export / Add / bulk-action buttons all render but are disabled with hover tooltips. Layout doesn't change when interactivity lands in slice 2 — only the disabled flags flip.
What slice 2 turns on
  • Add-drawing flow with band-aware suggestions, drawing edit form, bulk-rev-advance, IFA→IFC project promotion, save, Excel export. The workspace is the same screen — slice 2 just makes the buttons real.
